Release & publicação

O versionamento é dirigido pelo manifesto (AzureUtils/AzureUtils.psd1). O merge de develop → main cria a tag de release e publica na PowerShell Gallery automaticamente — sem necessidade de personal access token.

Fluxo de versionamento

  1. O trabalho acontece em develop. Suba o ModuleVersion (e PrivateData.PSData.Prerelease para pré-lançamentos) lá.
  2. O merge develop → main (que altera AzureUtils/AzureUtils.psd1) executa o Auto-tag release on main, que:
    • resolve a versão a partir do manifesto e cria a tag git — estável v{ModuleVersion}, pré-lançamento v{ModuleVersion}-{Prerelease} — apenas se ela ainda não existir;
    • dispara o workflow de publicação via gh workflow run usando o github.token embutido.
  3. O Publish PowerShell Gallery então roda um pipeline guard → validate → publish: publica apenas quando o commit da tag é alcançável a partir de main e a tag bate com a versão do manifesto. O passo de publicação é idempotente (uma versão já publicada é tratada como sucesso).
ℹ
Nenhum personal access token é necessário: a tag é enviada com o github.token padrão e o workflow de publicação é iniciado explicitamente via gh workflow run (ele também aceita workflow_dispatch com uma entrada release_tag para re-execuções manuais).

Secrets do repositório necessários

SecretUsado porNotas
PSGALLERY_API_KEYPublish PowerShell GalleryChave de API da PowerShell Gallery.

O workflow auto-tag precisa de permissions: contents: write e actions: write (já declarados no workflow) para poder enviar a tag e disparar o workflow de publicação com o token embutido.
