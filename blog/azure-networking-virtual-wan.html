Azure Virtual WAN: managed networking at scale

Azure Virtual WAN is Microsoft's managed answer for global-scale connectivity: virtual hubs that connect VNets, branches (VPN/SD-WAN), ExpressRoute and remote users, with centralized transitivity and routing. It is a managed Hub-and-Spoke. This is Part 3 of the series — compare it with the traditional Hub-Spoke from Part 2.

Henderson Andrade Published Jun 19, 2026 ~14 min read Azure · Virtual WAN · Global scale
Virtual WAN Virtual Hub Routing intent Series · 3/6

Executive summary

ℹ
Virtual WAN trades granular control for managed abstraction. Instead of building hub, peerings, gateways and route tables by hand, Microsoft delivers a managed Virtual Hub where VNets, VPN, ExpressRoute and P2S connect with native transitivity. It is a strong case for many branches, multiple regions and global connectivity. The price: its own routing model (hub route tables, labels, routing intent) with a learning curve, and less freedom for very specific NVA patterns. For small, regional environments it is usually overengineering.
⚠
Routing behavior, NVA integration and each Virtual WAN SKU's capabilities evolve — validate against the official documentation before adopting.

Concept

Virtual WAN is a resource that aggregates Virtual Hubs — Microsoft- managed hubs, one per region — to which you connect VNets (VNet connections), VPN/SD-WAN sites, ExpressRoute circuits and P2S users. The hubs interconnect over the backbone, giving global transitivity without you designing a peering mesh. Same hub-and-spoke idea, but with the hub and much of the routing delivered as a service.

Components

  • Virtual WAN — the umbrella resource that aggregates the hubs.
  • Virtual Hub — the managed per-region hub; hosts gateways and routing.
  • VNet connections — connect your VNets to the hub (replacing manual peering).
  • VPN Gateway / ExpressRoute Gateway / P2S Gateway — inside the hub, for sites, circuits and users.
  • Route tables & labels — the hub's routing model (association and propagation).
  • Routing intent & Secured Hub — policies that send traffic (Internet and/or private) to the Azure Firewall in the hub.

Routing intent and Secured Virtual Hub

The Secured Virtual Hub is a Virtual Hub with integrated Azure Firewall. With routing intent, you declare the intent to inspect Internet and/or private traffic (between VNets, branches and on-premises), and Virtual WAN programs the routes to flow through the firewall — without you building UDRs by hand in every spoke. It is the managed equivalent of the "UDR 0.0.0.0/0 to the firewall" from traditional Hub-Spoke (Part 2), but using the vWAN routing model.

ℹ
Not every NVA/inspection pattern from traditional Hub-Spoke behaves the same in Virtual WAN. If your organization depends on a specific NVA with a very particular routing model, validate compatibility with vWAN before deciding.

Virtual WAN vs. traditional Hub-Spoke

CriterionTraditional Hub-SpokeAzure Virtual WAN
ControlMore manual controlMore managed
OperationsMore customer responsibilityMore abstraction and automation
Global scaleRequires your own design (multiple hubs)More natural (inter-hub transitivity)
BranchesPossible, but adds complexityStrong use case
RoutingUDRs, peerings, gatewaysHub route tables, labels, routing intent
FirewallAzure Firewall/NVA in the hub VNetSecured Hub / Azure Firewall
Learning curveKnown by Azure teamsRequires understanding the vWAN model

The two are not mutually exclusive: it is common to have a traditional hub in one region and Virtual WAN for global connectivity, or vWAN with ExpressRoute and VPN backup (Part 4).

When to use and when to avoid

Use Virtual WAN when…Reconsider when…
There are many branches and VPN/SD-WAN sitesThe environment is small and regional
There are multiple regions and global connectivityYou need extreme control over each route
You want to simplify connectivity operationsYou depend on very specific NVAs/inspection models
You want standardized global transitivityThere is no maturity yet to operate the vWAN routing model

Checklist

  • Justification — Does the scale (branches/regions) really justify vWAN over Hub-Spoke?
  • Routing — Route tables, labels and routing intent designed and understood by the team?
  • Security — Secured Hub with Azure Firewall and routing intent for Internet/private?
  • Hybrid — ExpressRoute and VPN connected to the hubs with redundancy (Part 4)?
  • Compatibility — NVAs and inspection patterns validated in the vWAN model?
