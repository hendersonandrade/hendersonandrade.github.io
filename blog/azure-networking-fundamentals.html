Azure Networking Fundamentals

Before choosing between Hub-and-Spoke, Virtual WAN or hybrid connectivity, you must master the blocks every Azure topology is built from: VNet, subnets, NSG, routes, peering, gateway transit, firewall, Private Link and DNS. This is Part 1 of a series on Azure network architecture — the conceptual base the enterprise decisions in the next parts rest on.

Henderson Andrade Published Jun 19, 2026 ~16 min read Azure · Networking · Landing Zone
Azure Networking VNet · Subnet · NSG UDR · Peering Private Link · DNS Series · 1/6

Executive summary

ℹ
Network topology is one of the earliest and most durable decisions in an Azure Landing Zone — hard to change once dozens of subscriptions and VNets depend on it. This part sets the foundations: the VNet is the private perimeter; subnets segment; NSG/ASG filter; UDR redirects the traffic path (typically to a firewall); peering connects VNets over the backbone but is not transitive; gateway transit lets spokes use the hub's hybrid gateway; Private Link brings PaaS into the private network — and requires careful DNS. Parts 2 to 4 show how to combine these blocks into Hub-Spoke, Virtual WAN and hybrid connectivity; parts 5 and 6 cover DNS/security and operations/governance/FinOps. No model is universally correct — what changes is the trade-off between control, scale, cost and operational effort.
⚠
Capabilities, SKU limits, regional availability and prices change often. Whenever this article cites a specific behavior, validate it against the official Microsoft Learn documentation before designing for production. The numbers here are conceptual and do not replace the current reference.

Why network topology is a central decision

Azure networking is not "connecting VNets". Topology defines how workloads communicate, how applications reach PaaS services, how on-premises reaches the cloud, where traffic is inspected, how governance is applied and how connectivity cost grows. It is the fourth of the eight design areas of Azure Landing Zones — and perhaps the one with the most inertia: changing topology after the platform has grown is expensive and risky.

That is why good architecture separates platform from workloads: the shared connectivity services (firewall, gateways, DNS, Bastion) live in a connectivity subscription owned by the platform team; workloads land in their own subscriptions with inherited guardrails. This decision — central to the Cloud Adoption Framework — only makes sense once you understand the blocks below.

Virtual Network and subnets

The Azure Virtual Network (VNet) is the logical unit of private networking: an address space (one or more CIDR blocks) within which you create subnets. Resources in the same VNet talk by default; what crosses the VNet boundary depends on peering, gateways and routes. IP planning is the hardest decision to reverse: pick blocks that do not overlap with on-premises or with other VNets that will one day peer, and leave room to grow.

Some subnets have reserved names required by specific services — plan them into the IP design from the start:

Subnets also support delegation (handing the subnet to a service's control plane, such as App Service VNet Integration, AKS or managed databases). Plan dedicated subnets for Private Endpoints, App Service integration and AKS — each has its own size requirements.

Network Security Groups and ASGs

The Network Security Group (NSG) is a stateful 5-tuple filter applied to a subnet or a NIC. Rules have priority, direction (inbound/outbound) and can use service tags (e.g. Storage, AzureKeyVault) instead of IPs, plus Application Security Groups (ASG) to group NICs by role (e.g. web, db) and write rules per logical group rather than per IP.

ℹ
NSG is necessary but not sufficient. It does L3/L4 micro-segmentation, not application, FQDN, TLS or IDS/IPS inspection. For that you need a firewall (next sections). Use NSG for the segmentation baseline and a firewall for centralized inspection.

System routes and User Defined Routes

Azure creates system routes automatically (within the VNet, to the Internet, for peering and gateway prefixes). You override that behavior with User Defined Routes (UDR) in a route table associated with the subnet. The most common case is the default route 0.0.0.0/0 pointing to an Azure Firewall or NVA — forcing all egress through inspection:

Forced tunneling (sending even Internet egress to the firewall or to on-premises) is powerful but should not be applied indiscriminately: it adds latency, cost and the risk of breaking platform dependencies. Use propagate gateway routes deliberately and validate the effective routes before trusting the design.

VNet Peering and Gateway Transit

VNet Peering connects two VNets over the Microsoft backbone (regional or global, across regions), with low latency and no gateway. The point that confuses most people: peering is not transitive. If spoke A peers with the hub and spoke B also peers with it, A does not reach B automatically — it requires explicit routing (via an NVA/ firewall in the hub, direct peering, or Virtual WAN). The flags that govern this: allow forwarded traffic, allow gateway transit and use remote gateway.

ℹ
Gateway transit solves a classic problem: the hybrid gateway (VPN/ExpressRoute) is expensive and lives in the hub. With allow gateway transit on the hub and use remote gateway on the spoke, spokes use the hub's gateway without owning one. Caveats: a spoke uses one remote gateway at a time, and route propagation must be planned so on-premises sees the spokes and vice-versa.

Azure Firewall vs. third-party NVA

Centralized inspection — the "single point" through which north-south and east-west traffic passes — can be done with Azure Firewall (native PaaS, managed, autoscaling, network and application rules, FQDN tags, Threat Intelligence) or with a third-party Network Virtual Appliance (NVA) (Palo Alto, Fortinet, Check Point...). The choice is less about isolated features and more about who operates it and what already exists in the organization:

⚠
A private network and a firewall are not synonymous with Zero Trust, and ExpressRoute does not replace encryption or identity control. Network security is one layer — not the only one.

Private Link, Private Endpoints and DNS

The Private Endpoint (PE) brings a PaaS service (Storage, SQL, Key Vault...) into your network with a private IP, removing public exposure. The detail that breaks most environments: the service FQDN stays the same, but it must resolve to the private IP — and that depends on correct DNS (an Azure Private DNS Zone linked to the VNet, with the PE record). Bad DNS resolves to the public endpoint and the PE becomes an IP nobody uses.

There are also Service Endpoints, older: they give the subnet an optimized route to the PaaS service over the backbone, but without a private IP and without Private Link's granularity. In restrictive architectures, the Private Endpoint is usually the preferred default:

CriterionPrivate EndpointService Endpoint
AccessPrivate IP in your VNetService's public IP, via optimized route
ExposureRemoves the public endpoint (with the service firewall)Keeps the public endpoint
DNSRequires a Private DNS ZoneNot required
Hybrid (on-prem)Reachable via private DNS/routesDoes not extend to on-premises
GranularityPer resourcePer subnet/service

NAT Gateway, Bastion, DDoS and Network Watcher

  • NAT Gateway — outbound egress with a stable, managed public IP for a subnet, without exposing inbound. Unlike the firewall: NAT Gateway does not inspect, it only does outbound SNAT.
  • Azure Bastion — administrative access (RDP/SSH) to VMs without public IPs on the VMs and without exposing those ports to the Internet. Lives in AzureBastionSubnet.
  • DDoS Protection — protection against volumetric attacks for VNets with critical publicly exposed resources. Evaluate it where the public surface is significant.
  • Network Watcher — the troubleshooting Swiss army knife: next hop, effective routes, effective security rules, connection troubleshoot, packet capture and topology. Without observability the network is a black box (Part 6 digs deeper).

Common pitfalls (fundamentals)

  • Overlapping address space between Azure and on-premises (or between VNets that will peer) — it breaks routing and is very expensive to fix later.
  • Assuming peering is transitive and expecting spokes to talk to each other with no route.
  • Creating a Private Endpoint without configuring the Private DNS Zone — the FQDN resolves to the public IP and the PE is unused.
  • Applying indiscriminate forced tunneling and breaking platform dependencies, or adding needless latency/cost.
  • Treating NSG as the only security control, with no application/FQDN inspection.
  • Not validating effective routes before trusting the routing design.

Fundamentals checklist

  • Addressing — IP plan with no overlap with on-premises and room to grow?
  • Subnets — Reserved (Gateway/Firewall/Bastion) and dedicated (PE/AKS/App Service) subnets planned?
  • Segmentation — NSG/ASG at the baseline and firewall/NVA for centralized inspection?
  • Routing — UDRs documented, propagation controlled, effective routes validated?
  • Connectivity — Peering and gateway transit designed knowing peering is not transitive?
  • Private PaaS — Private Endpoints with Private DNS Zones linked to the right VNets?
  • Operations — Network Watcher and logs enabled from day 1?
