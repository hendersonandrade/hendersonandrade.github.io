Azure Virtual WAN: rede gerenciada em escala

O Azure Virtual WAN é a resposta gerenciada da Microsoft para conectividade em escala global: hubs virtuais que conectam VNets, filiais (VPN/SD-WAN), ExpressRoute e usuários remotos, com transitividade e roteamento centralizados. É um Hub-and-Spoke gerenciado. Esta é a Parte 3 da série — compare com o Hub-Spoke tradicional da Parte 2.

Resumo executivo

ℹ
O Virtual WAN troca controle granular por abstração gerenciada. Em vez de você construir hub, peerings, gateways e route tables à mão, a Microsoft entrega um Virtual Hub gerenciado onde VNets, VPN, ExpressRoute e P2S se conectam com transitividade nativa. É um caso forte para muitas filiais, múltiplas regiões e conectividade global. O preço: um modelo de roteamento próprio (route tables do hub, labels, routing intent) com sua curva de aprendizado, e menos liberdade para padrões de NVA muito específicos. Para ambientes pequenos e regionais, costuma ser overengineering.
⚠
O comportamento de roteamento, a integração de NVAs e as capacidades de cada SKU do Virtual WAN evoluem — valide na documentação oficial antes de adotar.

Conceito

O Virtual WAN é um recurso que agrega Virtual Hubs — hubs gerenciados pela Microsoft, um por região — aos quais você conecta VNets (VNet connections), sites VPN/SD-WAN, circuitos ExpressRoute e usuários P2S. Os hubs se interligam pelo backbone, dando transitividade global sem você desenhar peerings em malha. É a mesma ideia de hub-and-spoke, mas com o hub e boa parte do roteamento como serviço.

Componentes

  • Virtual WAN — o recurso "guarda-chuva" que agrega os hubs.
  • Virtual Hub — o hub gerenciado por região; hospeda gateways e o roteamento.
  • VNet connections — conectam suas VNets ao hub (substituem o peering manual).
  • VPN Gateway / ExpressRoute Gateway / P2S Gateway — dentro do hub, para sites, circuitos e usuários.
  • Route tables & labels — o modelo de roteamento do hub (associação e propagação).
  • Routing intent & Secured Hub — políticas que enviam tráfego (Internet e/ou privado) ao Azure Firewall no hub.

Routing intent e Secured Virtual Hub

O Secured Virtual Hub é um Virtual Hub com Azure Firewall integrado. Com routing intent, você declara a intenção de inspecionar o tráfego de Internet e/ou o tráfego privado (entre VNets, branches e on-premises), e o Virtual WAN programa as rotas para passar pelo firewall — sem você montar UDRs manualmente em cada spoke. É o equivalente gerenciado ao "UDR 0.0.0.0/0 para o firewall" do Hub-Spoke tradicional (Parte 2), porém com o modelo de rotas do vWAN.

ℹ
Nem todo padrão de NVA/inspeção do Hub-Spoke tradicional se comporta igual no Virtual WAN. Se a sua organização depende de uma NVA específica com um modelo de roteamento muito particular, valide a compatibilidade com o vWAN antes de decidir.

Virtual WAN vs. Hub-Spoke tradicional

CritérioHub-Spoke tradicionalAzure Virtual WAN
ControleMaior controle manualMais gerenciado
OperaçãoMais responsabilidade do clienteMais abstração e automação
Escala globalExige desenho próprio (múltiplos hubs)Mais natural (transitividade entre hubs)
FiliaisPossível, mas aumenta a complexidadeCaso de uso forte
RoteamentoUDRs, peerings, gatewaysRoute tables do hub, labels, routing intent
FirewallAzure Firewall/NVA na VNet do hubSecured Hub / Azure Firewall
Curva de aprendizadoConhecida por times AzureRequer entender o modelo vWAN

Os dois não são excludentes: é comum um hub tradicional em uma região e Virtual WAN para conectividade global, ou vWAN com ExpressRoute e VPN backup (Parte 4).

Quando usar e quando evitar

Use Virtual WAN quando…Reavalie quando…
Há muitas filiais e sites VPN/SD-WANO ambiente é pequeno e regional
Há múltiplas regiões e conectividade globalVocê precisa de controle extremo sobre cada rota
Quer simplificar a operação de conectividadeDepende de NVAs/modelos de inspeção muito específicos
Quer transitividade global padronizadaAinda não há maturidade para operar o modelo de rotas do vWAN

Checklist

  • Justificativa — A escala (filiais/regiões) realmente justifica o vWAN sobre o Hub-Spoke?
  • Roteamento — Route tables, labels e routing intent desenhados e entendidos pelo time?
  • Segurança — Secured Hub com Azure Firewall e routing intent para Internet/privado?
  • Híbrido — ExpressRoute e VPN conectados aos hubs com redundância (Parte 4)?
  • Compatibilidade — NVAs e padrões de inspeção validados no modelo do vWAN?
