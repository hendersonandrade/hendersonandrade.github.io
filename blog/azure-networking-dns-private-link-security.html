DNS, Private Link and network security

Topology connects; DNS makes names resolve to the right places; security controls who talks to whom. This Part 5 digs into the two topics that most often break (or save) private architectures: hybrid DNS with the DNS Private Resolver and layered security — closing with Zero Trust applied to the network.

Henderson Andrade Published Jun 19, 2026 ~15 min read Azure · DNS · Private Link · Security
DNS Private Resolver Private Endpoints Azure Firewall · WAF Series · 5/6

Executive summary

ℹ
DNS is the biggest source of error in private Azure architectures: a Private Endpoint without the right Private DNS Zone resolves to the public IP and privacy evaporates. The DNS Private Resolver replaces the old forwarder VMs, connecting Azure DNS to on-premises both ways. In security, think in layers: segmentation (VNet/subnet/NSG/ASG), centralized inspection (Azure Firewall/NVA), application protection (WAF on Application Gateway or Front Door), private PaaS access (Private Endpoints) and egress control. And remember: a private network is not Zero Trust — Zero Trust requires identity, least privilege, inspection and continuous validation, not just a private IP.
⚠
Each SKU's features (e.g. Azure Firewall premium capabilities) and DNS behaviors change — validate against the official documentation before designing.

Hybrid DNS and the DNS Private Resolver

The Azure DNS Private Resolver handles hybrid DNS without forwarder VMs. It has inbound endpoints (on-premises queries Azure private names) and outbound endpoints (Azure workloads forward queries to on-premises DNS, via conditional forwarding rules). The typical flow:

  • 1 — The on-premises client queries its local DNS.
  • 2 — Local DNS forwards the Azure private zone to the Private Resolver's inbound endpoint.
  • 3 — Azure workloads query the DNS configured on the VNet.
  • 4 — The Private Resolver forwards (conditional forwarding) on-premises names to local DNS via the outbound endpoint.
  • 5 — The Private Endpoint resolves to the private IP via the Private DNS Zone (Part 1).

Common DNS pitfalls

  • Creating a Private Endpoint without configuring DNS — the FQDN resolves to the public IP.
  • VNet not linked to the service's Private DNS Zone.
  • On-premises DNS not forwarding Azure private zones.
  • Duplicating Private DNS Zones per subscription without governance (centralize or standardize).
  • Forgetting the FQDN stays the same — only resolution changes to the private IP.
  • Mixing custom DNS and Azure-provided DNS without a clear design.

Firewall, WAF and egress/ingress control

Don't confuse the roles: Azure Firewall inspects network traffic (north-south and east-west); the WAF protects web applications — on Application Gateway (regional protection) or Azure Front Door (global edge protection). Egress control usually flows through the firewall with FQDN rules; ingress through Application Gateway/Front Door with WAF. North-south inspection is traffic in/out; east-west is between workloads (spoke-to-spoke from Part 2).

Zero Trust applied to the network

⚠
A private network is not synonymous with Zero Trust. Zero Trust assumes strong identity, least privilege, segmentation, inspection, monitoring, continuous validation, reduced public exposure and egress control. The network is one layer — combine it with identity (Entra ID) and governance (Part 6).

Checklist

  • DNS — Private DNS Zones centralized/governed and the right VNets linked?
  • Hybrid — On-premises resolves Azure private names and vice-versa (Private Resolver)?
  • Layers — Segmentation + NSG + firewall + WAF + Private Endpoints in place?
  • Egress/Ingress — Egress inspected by the firewall; ingress protected by WAF?
  • Zero Trust — Identity, least privilege and continuous validation, not just a private IP?
