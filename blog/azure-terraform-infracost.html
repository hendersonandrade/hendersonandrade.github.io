Azure + Terraform + Infracost: enterprise IaC with FinOps in the pipeline

An end-to-end lab: from the concepts of Infrastructure as Code and FinOps to a modular, professional Terraform structure on Azure — with a state backend bootstrapped by Bicep, a GitHub Actions pipeline authenticated through secretless OIDC and Infracost cost estimates automatically commented on every Pull Request. All the code lives in a companion repository you can clone and run.

Henderson Andrade Published Jun 18, 2026 ~24 min read Terraform · Azure · Infracost · GitHub Actions
Terraform FinOps · Infracost Bicep backend OIDC / WIF GitHub Actions 6 diagrams
ℹ
Prerequisites: an Azure subscription (Owner/Contributor), Azure CLI, Terraform CLI ≥ 1.7, a GitHub repository and a free Infracost account. This article focuses on the concepts and architecture decisions; the full step-by-step guide to stand the pipeline up — bootstrap, OIDC, Variables/Secrets, opening a PR, applying and destroying — lives in the azure-iac-with-tf-infracost repository README, complete with a prerequisites checklist and troubleshooting.

What Infrastructure as Code really is

Infrastructure as Code (IaC) is the practice of describing your infrastructure — networks, machines, databases, permissions — in versioned text files, and letting a tool provision it for you. Instead of clicking through the portal and hoping you remember what you did, you declare the desired state and the tool reconciles reality with that target.

Four concepts hold up everything that follows:

  • Declarative > imperative. You describe what you want (a Standard_B2s VM in this subnet), not the steps to get there. Terraform figures out the how — create, update or destroy — by comparing desired with actual.
  • Idempotency. Running the same code ten times yields the same result. If nothing changed in the code, apply does nothing. That makes automation safe to repeat.
  • State. Terraform keeps a state file that maps each resource in code to the real resource in Azure. It is the tool's source of truth — which is why it must be stored remotely, securely and with locking (more on that soon).
  • Drift & immutability. Drift is when someone changes something by hand and reality diverges from code. plan detects drift; apply corrects it. The immutable mindset prefers to recreate over patch — servers are cattle, not pets.

Why it matters. IaC brings repeatability (dev is identical to prod), versioning (every change is a commit with an author and a reason), review (infra changes via Pull Request, with peer review), speed (standing up a new environment is running a pipeline) and governance (policies and standards become code, not tribal knowledge). It is the eighth design area of Azure Landing Zones — platform automation — taken down to the workload level.

FinOps and the invisible-cost problem

In the cloud, every line of IaC carries a price. Swapping Standard_B2s for Standard_D8s_v5 is a one-word diff that multiplies the bill tenfold. The trouble is that, in the traditional flow, nobody sees that cost until the invoice arrives — by which point the resource has already run for a whole month.

FinOps is the discipline that brings financial accountability to cloud speed: engineering, finance and the business collaborating so each team sees and owns its own cost. The FinOps Foundation organizes it into three iterative phases:

PhaseQuestion it answersIn our lab
InformHow much does it cost and whose is it?Infracost shows the diff's cost on the PR; tags identify the owner.
OptimizeWhere can we save?Compare SKUs in the plan before applying; right-sizing.
OperateHow do we keep it in check?Infracost guardrails/policies that fail expensive PRs.

The big idea is shift-left of cost: pushing financial visibility to the left of the pipeline — into the Pull Request, where the change is still cheap to adjust — instead of discovering it on the invoice, where it is already too late.

Infracost in depth

The Infracost CLI is open source (Apache 2.0) and estimates the monthly cost of your Terraform infrastructure before you apply it. It reads your code (or the generated plan), identifies each billable resource, queries the cloud pricing database and returns a breakdown (total cost) or a diff (how much this PR adds or removes). Worth the distinction: the CLI is open source, but it is part of a commercial product — pricing uses the hosted Cloud Pricing API (free, requires an API key/sign-up) and Infracost Cloud is a commercial SaaS (with a free tier).

How it works, under the hood:

  1. Parse. Infracost reads the Terraform project (HCL) or a terraform plan in JSON and extracts the resources and their price-relevant parameters (SKU, tier, region, disk size).
  2. Pricing. For each resource, it queries Infracost's Cloud Pricing API — a catalog of the public prices for AWS, Azure and GCP — and computes the projected monthly cost.
  3. Output. It emits a table (breakdown) or a comparison (diff) as text, JSON, HTML or a Pull Request comment.

Infracost Cloud is the SaaS dashboard (with a free tier) that aggregates these numbers across repositories, shows trends and, crucially, lets you define guardrails/policies — for example, "flag for review any PR that increases cost by more than US$ 100/mo." The advantages are clear: cost visible on the PR, informed SKU decisions, prevention of invoice surprises and a FinOps culture baked into the pipeline. The limitations to keep in mind: it is a list-price estimate (it does not account for Reserved Instance/Savings Plan discounts or actual hourly usage), and consumption-based resources (egress, requests) depend on usage assumptions you can tune in infracost-usage.yml.

The enterprise IaC structure

An 800-line main.tf works until the day two teams need to touch it. The enterprise approach is modular: small, reusable modules, a root that composes them, and per-environment variables. This is the companion repo's tree:

azure-iac-with-tf-infracost/
# state backend bootstrap — Bicep (one-time run)
bootstrap/
  main.bicep            # RG + Storage Account + "tfstate" container
  modules/state-storage.bicep
  deploy.sh

# Terraform root applied by the pipeline
infra/
  providers.tf         # versions + remote "azurerm" backend (OIDC)
  main.tf              # composes the modules
  variables.tf  outputs.tf  data.tf
  dev.tfvars    prod.tfvars

# reusable modules (1 responsibility each)
modules/
  networking/          # VNet + subnet + NSG
  storage-account/     # 1 Storage Account
  virtual-machine/     # 1 Linux VM + NIC
  app-service/         # 1 App Service Plan + Web App

infracost/infracost.yml
.github/workflows/terraform.yml

Each module exposes a clear contract — variables.tf (inputs), main.tf (resources) and outputs.tf (outputs) — and nothing else. The naming convention is <resource>-<workload>-<environment> (e.g. vnet-iacdemo-dev), and common tags (environment, managedBy, costCenter) are merged into every resource — essential for FinOps to attribute cost to the right owner.

Why the tfstate is provisioned with Bicep

Terraform stores its state in a backend. In production that backend is remote — on Azure, a Storage Account + container — so the state is shared by the team and the pipeline, encrypted at rest and locked (preventing two concurrent apply runs from corrupting everything).

But here comes the chicken-and-egg problem: Terraform needs the backend to run, but the backend is... infrastructure. If you create the Storage Account with Terraform, where does the state that describes that Storage Account live? In a throwaway local state nobody else has? In a state that references itself? Every path is fragile.

ℹ
The clean solution is to bootstrap the backend with Bicep. Bicep is native to Azure (it compiles to ARM), it is idempotent and — crucially — it keeps no local state: its "state" is Resource Manager itself. You run Bicep once to create the backend, and from then on all Terraform uses it. The bootstrap sits outside the Terraform lifecycle, exactly where it belongs.
bootstrap/modules/state-storage.bicep
resource sa 'Microsoft.Storage/storageAccounts@2023-05-01' = {
  name: storageAccountName
  location: location
  sku: { name: 'Standard_LRS' }
  kind: 'StorageV2'
  properties: {
    minimumTlsVersion: 'TLS1_2'
    allowBlobPublicAccess: false
    supportsHttpsTrafficOnly: true
  }
}

resource blobService 'Microsoft.Storage/storageAccounts/blobServices@2023-05-01' = {
  parent: sa
  name: 'default'
  properties: {
    isVersioningEnabled: true          // recover overwritten state
    deleteRetentionPolicy: { enabled: true, days: 30 }
  }
}
ℹ
Run the bootstrap once with bash bootstrap/deploy.sh (which calls az deployment sub create). Note the resourceGroupName and storageAccountName outputs — they go into the GitHub Variables so terraform init can configure the backend.

Workload Identity Federation: a secretless pipeline

The old way to authenticate a pipeline to Azure was to create a service principal with a client secret and store that password as a GitHub secret. The problem: it is a long-lived credential that leaks, expires and needs rotation. The modern way is OpenID Connect (OIDC), also called Workload Identity Federation.

The idea: GitHub Actions issues a short-lived OIDC token for each run, proving "I am repository X, on branch Y." In Entra ID you configure a Federated Credential that trusts that token for a specific subject (repo:org/repo:ref:refs/heads/main). Azure exchanges the GitHub token for an access token — no password is stored anywhere.

create the federated credential (Azure CLI)
az ad app federated-credential create --id "$appId" --parameters '{
  "name": "gh-main",
  "issuer": "https://token.actions.githubusercontent.com",
  "subject": "repo:hendersonandrade/azure-iac-with-tf-infracost:ref:refs/heads/main",
  "audiences": ["api://AzureADTokenExchange"]
}'

The Terraform samples, annotated

providers.tf pins versions and declares the remote backend. Note use_oidc = true — that is what wires everything to the secretless login from the previous section. The backend values that vary (resource group, storage account) cannot use variables, so they are passed at init time via -backend-config:

infra/providers.tf
terraform {
  required_version = ">= 1.7.0"
  required_providers {
    azurerm = { source = "hashicorp/azurerm", version = "~> 4.0" }
  }
  backend "azurerm" {
    container_name = "tfstate"
    key            = "azure-iac-with-tf-infracost.tfstate"
    use_oidc       = true   # backend authenticates with the OIDC token
  }
}

provider "azurerm" {
  features {}
  use_oidc = true           # no secret: login via WIF
}

A module always follows the same contract. The storage one uses a random suffix to guarantee a globally unique name:

modules/storage-account/main.tf
resource "azurerm_storage_account" "this" {
  name                            = local.sa_name
  resource_group_name             = var.resource_group_name
  location                        = var.location
  account_tier                    = var.account_tier      # Standard
  account_replication_type        = var.replication_type  # LRS
  min_tls_version                 = "TLS1_2"
  allow_nested_items_to_be_public = false
  https_traffic_only_enabled      = true
  tags                            = var.tags

  blob_properties {
    versioning_enabled = true
  }
}

And the root composes everything, passing one module's outputs as another's inputs — note the subnet_id from the networking module flowing into the VM, and the use of a data source (azurerm_subscription) to read metadata without creating anything:

infra/main.tf · data.tf
# data.tf — reads what already exists, creates nothing
data "azurerm_client_config" "current" {}
data "azurerm_subscription" "current" {}

# main.tf — the VM consumes the subnet_id the network exposed
module "networking" {
  source        = "../modules/networking"
  name_suffix   = local.name_suffix
  address_space = var.address_space
  # ...
}

module "vm" {
  source    = "../modules/virtual-machine"
  subnet_id = module.networking.subnet_id   # output → input
  vm_size   = var.vm_size
  # ...
}

The GitHub Actions pipeline, step by step

The workflow has two jobs: plan (runs on every PR and push, validates and estimates cost) and apply (only on push to main, behind an environment with manual approval). Let's walk through the blocks.

1 · Triggers and permissions. The permissions block is the heart of OIDC: id-token: write authorizes the job to request the token. pull-requests: write lets Infracost comment on the PR.

.github/workflows/terraform.yml — trigger & permissions
on:
  pull_request: { branches: [main] }
  push:         { branches: [main] }

permissions:
  id-token: write        # request the OIDC token
  contents: read         # checkout the code
  pull-requests: write   # comment cost on the PR

2 · OIDC login + init with backend. The azure/login action receives only the identifiers (client/tenant/subscription) — no password — and uses the OIDC token. Then init receives the Bicep backend details via -backend-config.

.github/workflows/terraform.yml — login & init
- name: Azure login (OIDC)
  uses: azure/login@v2
  with:
    client-id:       ${{ vars.AZURE_CLIENT_ID }}
    tenant-id:       ${{ vars.AZURE_TENANT_ID }}
    subscription-id: ${{ vars.AZURE_SUBSCRIPTION_ID }}

- name: Terraform init
  run: |
    terraform init \
      -backend-config="resource_group_name=${TF_BACKEND_RG}" \
      -backend-config="storage_account_name=${TF_BACKEND_SA}"

3 · fmt → validate → plan. Three gates in increasing order of cost: formatting (style), validation (syntax and references) and plan, which becomes an artifact reused by apply — guaranteeing that what was reviewed is what gets applied.

.github/workflows/terraform.yml — validation & plan
- run: terraform fmt -check -recursive
- run: terraform validate
- run: terraform plan -input=false -var-file="dev.tfvars" -out=tfplan

- name: Upload plan artifact
  uses: actions/upload-artifact@v4
  with: { name: tfplan, path: infra/tfplan }

4 · Protected apply. The apply job only fires on push to main and declares environment: production — which, under Settings → Environments on GitHub, enables the requirement that reviewers approve before the deploy. It is your human gate over the automation.

Integrating Infracost into the pipeline

With the pipeline in place, wiring in Infracost is three steps, only in the Pull Request context: install the CLI, generate the cost diff and post/update the PR comment.

.github/workflows/terraform.yml — Infracost on the PR
- name: Setup Infracost
  if: github.event_name == 'pull_request'
  uses: infracost/actions/setup@v3
  with: { api-key: ${{ secrets.INFRACOST_API_KEY }} }

- name: Infracost diff
  if: github.event_name == 'pull_request'
  run: |
    infracost diff \
      --config-file=../infracost/infracost.yml \
      --format=json --out-file=/tmp/infracost.json

- name: Comment on PR
  if: github.event_name == 'pull_request'
  run: |
    infracost comment github \
      --path=/tmp/infracost.json \
      --repo=$GITHUB_REPOSITORY \
      --github-token=${{ secrets.GITHUB_TOKEN }} \
      --pull-request=${{ github.event.pull_request.number }} \
      --behavior=update

--behavior=update keeps a single comment that is updated on every push to the PR — instead of cluttering the thread with a new one per commit. The infracost/infracost.yml file defines which projects/var-files to cost (dev and prod), so the reviewer sees the impact across both environments.

ℹ
Guardrails. In Infracost Cloud (free tier) you enable policies that flag or block PRs above a threshold — e.g. "an increase > US$ 100/mo requires FinOps approval." It is the FinOps Operate phase turning into an automatic gate on the pipeline.

Production checklist

  • State — Remote backend on Azure Storage, with versioning and locking, created by a separate Bicep bootstrap?
  • Authentication — Pipeline via OIDC/Workload Identity Federation, with no client secret stored anywhere?
  • Modularity — Resources in reusable modules with a clear contract (variables/outputs), not a monolith?
  • Environments — Per-environment config isolated in *.tfvars (dev/prod), same code?
  • CI quality — fmt -check, validate and plan running on every PR, with the plan becoming an artifact?
  • Human gate — apply only on main, behind an environment with mandatory approval?
  • FinOps — Infracost commenting the cost diff on the PR, with a threshold guardrail in Infracost Cloud?
  • Tags — environment, managedBy and costCenter on every resource for cost attribution?
  • Lifecycle — A documented destroy path so ephemeral environments don't become forgotten cost?
