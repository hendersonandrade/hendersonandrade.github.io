Hub-and-Spoke Architecture on Azure

The traditional Hub-and-Spoke is Azure's best-known enterprise topology: a central VNet (the hub) concentrates firewall, gateways, DNS and Bastion; each workload lives in an isolated spoke VNet peered to the hub. This is Part 2 of the series — built on the Fundamentals from Part 1.

Henderson Andrade Published Jun 19, 2026 ~14 min read Azure · Hub-Spoke · Landing Zone
Hub-and-Spoke Peering · Gateway transit Azure Firewall Series · 2/6

Executive summary

ℹ
In traditional Hub-Spoke, you own the routing: the hub hosts shared services (firewall/NVA, VPN/ExpressRoute Gateway, DNS, Bastion), spokes host workloads, and peering + UDRs stitch it together. It is the model with the most granular control and the closest fit to traditional Landing Zones. The price of that control is operations: many peering relationships, route tables and inspection paths to manage as the number of VNets and regions grows. When global scale and branch count explode, Virtual WAN (Part 3) tends to simplify.
⚠
Sizing, peering limits per VNet and SKU capabilities change — always validate against the official documentation before designing for production.

Concept

The topology uses a central VNet — the hub — to host shared connectivity, security, DNS and management services. The spoke VNets host workloads and connect to the hub via VNet Peering. The hub usually lives in the connectivity subscription (owned by the platform); each spoke usually lives in its workload's subscription. As we saw in Part 1, peering is not transitive — so spoke-to-spoke and hybrid traffic depend on explicit routing through the hub.

Hub and spoke components

In the hub (shared)In each spoke (workload)
Azure Firewall or NVA (inspection)Application and data subnets

The eight traffic flows

Designing Hub-Spoke is, in practice, designing these eight paths — each with its route and firewall rule:

  • 1 · Spoke → Internet — UDR 0.0.0.0/0 sends to the hub firewall, which inspects and egresses.
  • 2 · Spoke → on-premises — via gateway transit; the route to on-prem prefixes points to the hub gateway.
  • 3 · On-premises → spoke — on-prem must see the spoke prefixes (route/BGP propagation).
  • 4 · Spoke → spoke — not automatic; goes through the hub firewall (or direct peering). See below.
  • 5 · Spoke → PaaS via Private Endpoint — resolution via Private DNS Zone to a private IP (Parts 1 and 5).
  • 6 · Remote user → workload — P2S on the gateway, or a modern/Zero Trust access solution.
  • 7 · Admin traffic → VM — Azure Bastion in the hub, no public IP on the VMs.
  • 8 · Published application — ingress via Application Gateway (regional WAF) or Azure Front Door (global edge WAF).

Spoke-to-spoke traffic: the options

The path between two spokes is the decision that sparks the most debate. The options, with trade-offs:

OptionProsCons
Via firewall/NVA in the hubCentralized east-west inspectionExtra latency and cost; firewall becomes a bottleneck
Direct spoke-to-spoke peeringLower latencyNo inspection; the peering mesh grows
Private Link between servicesGranular, no exposing the whole VNetApplies to services that support it
Avoid by designLess coupling between workloadsNot always possible

Pros and cons

  • + Granular control — you decide every route, peering and inspection path.
  • + Firewall flexibility — Azure Firewall or the corporate NVA the team already runs.
  • + Alignment — fits traditional Landing Zones and platform/workload separation.
  • − Operations grow — many peerings, routes and paths to maintain by hand.
  • − Multi-region — requires multiple hubs and inter-hub connectivity you design yourself.
  • − Branch scale — connecting many VPN/SD-WAN sites gets laborious (a strong Virtual WAN case).

When to use and when to avoid

Use Hub-Spoke when…Reconsider when…
You need high routing controlThere are many branches and complex global connectivity
The number of VNets is moderateMany VNets and regions with global transitivity
You depend on specific NVAsPeering/route operations become too manual
Regional or controlled multi-region architectureYou want a more managed connectivity service

Checklist

  • Separation — Hub in the connectivity subscription, spokes in workload subscriptions?
  • Gateway transit — Enabled on the hub and use remote gateway on the spokes?
  • Inspection — Spoke UDR 0.0.0.0/0 to the firewall, with effective routes validated?
  • Spoke-to-spoke — Path chosen deliberately (firewall vs. direct peering vs. Private Link)?
  • Multi-region — A plan for per-region hubs and inter-hub connectivity, if applicable?
