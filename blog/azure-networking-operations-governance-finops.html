Network as a platform: operations, governance and FinOps

A well-designed topology that nobody observes, governs or measures becomes an expensive black box. This Part 6 closes the series by treating the network as a product/platform: observability, governance with IaC and Policy as Code, network FinOps, the antipatterns to avoid and — finally — the decision table that ties together everything from Parts 1 to 5.

Henderson Andrade Published Jun 19, 2026 ~16 min read Azure · Operations · Governance · FinOps
Observability IaC · Policy as Code Network FinOps Series · 6/6

Executive summary

ℹ
Treat the network as a platform: reusable patterns, automation and clear change processes. Observability (Network Watcher, logs in Log Analytics/Sentinel) takes the network out of the black box. Governance comes from IaC (Terraform/Bicep), Policy as Code (Azure Policy) and RBAC that separates platform from workloads. FinOps is crucial because networking generates often-underestimated costs — firewall, gateways, logs and traffic. And on choosing the model: none is universally correct, and they combine (Hub-Spoke with ExpressRoute, vWAN with VPN backup, etc.).
⚠
Network costs (firewall, gateways, traffic, logs, Private Endpoints) vary by region, volume and SKU and change over time. Use Infracost and the official calculator; do not trust prices from memory. (See the Terraform + Infracost article for cost on the PR.)

Observability and troubleshooting

  • Network Watcher — next hop, effective routes, effective security rules, connection troubleshoot, packet capture and topology.
  • Logs — Azure Firewall logs, NSG flow logs, VPN Gateway diagnostics, ExpressRoute metrics, Application Gateway/Front Door logs, Private Endpoint state.
  • Metrics — throughput, latency, packet drops, tunnel/BGP status, SNAT utilization, firewall health, gateway utilization, data processed, DNS errors.
  • SIEM — send to Log Analytics and Microsoft Sentinel to detect anomalous traffic, unauthorized egress, unexpected cross-segment communication and changes to critical routes/rules.

Governance and automation

  • IaC — the whole network in Terraform/Bicep, with reusable modules, pull requests, architectural review and pre-deploy validation.
  • Policy as Code — Azure Policy and initiatives to block public IP on sensitive resources, require Private Endpoint, restrict regions, audit NSGs/diagnostics, enforce tags and prevent creating network resources outside the connectivity subscription.
  • RBAC — platform manages hub/connectivity; application teams manage workloads within limits; security audits. Prevent apps from creating public routes or firewall bypass.
  • Naming & tagging — the base for operations, FinOps, troubleshooting and automation.
  • Controlled change — changes to routes, firewall, DNS and peering can cause wide outages: plan, window, tests, rollback and effective-routes validation.

Network FinOps

Networking is a large and often underestimated cost. The main drivers and the decisions that impact them:

Cost driverDecision that impacts it
Azure Firewall / NVACentralizing all traffic without weighing cost/latency
Gateways (VPN/ExpressRoute) / circuitDuplicating gateways per region without analysis
VNet Peering & data transferCross-region traffic and inefficient routing
Private EndpointsMassive use without governance
Logs (Log Analytics)Overly verbose logs without proper retention
DDoS / WAF / Public IPsEnabling where there is no surface to justify it

FinOps recommendations: measure traffic, estimate growth, classify workloads by criticality, define per-criticality standards, review logs/retention, use tagging and track costs by subscription, resource group, application and service — avoiding overengineering.

Common antipatterns

  • Isolated VNets with no connectivity strategy; mesh peerings with no governance.
  • Each application creating its own firewall, gateway and DNS; public IP on everything "for simplicity".
  • Private Endpoints without correct private DNS; centralizing everything on the firewall without weighing cost/latency.
  • Not documenting routes and flows; not using IaC; manual changes to critical route tables.
  • Not monitoring gateways, tunnels, BGP and firewall; mixing Hub-Spoke and Virtual WAN with no clear architecture.
  • ExpressRoute without a VPN contingency when continuity is required; VPN as a critical link without weighing latency.
  • Underestimating IP planning; overlapping addressing; ignoring hybrid DNS.
  • Not separating platform and workloads; NSG as the only control; ignoring log and traffic costs.

Remember: they are not mutually exclusive. Hub-Spoke with ExpressRoute and VPN backup; vWAN with Secured Hub, ExpressRoute and VPN; Private Endpoints in spokes with centralized DNS — all coexist.

Practical scenarios (summary)

ScenarioLikely recommendation
Small company starting outSimple Hub-Spoke (or single VNet); S2S VPN if on-prem; avoid overengineering
Mid-size, multiple appsHub-Spoke with firewall/VPN/ExpressRoute/DNS; spokes per domain; IaC and policies
Regulated, central securityHub-Spoke with Firewall/NVA, forced tunneling, Private Endpoints, central DNS, SIEM
Many branchesVirtual WAN; VPN/SD-WAN sites; Secured Hub; ExpressRoute where critical
Global multi-regionVirtual WAN or multiple hubs; weigh latency/traffic/cost; Front Door for web apps
Datacenter migrationExpressRoute primary + VPN backup; Hub-Spoke; hybrid DNS; intensive monitoring
Cloud-native (PaaS)Private Endpoints; Private DNS Zones; egress control; less IaaS
Dev/labCheap architecture; simple VPN or no hybrid; minimal anti-exposure policies
ℹ
End of the series. You went through the fundamentals, the three big models, hybrid connectivity, DNS/security and now operations/governance/FinOps. The goal was not to memorize "what each architecture is", but to know when to choose each model, which trade-offs to weigh and which mistakes to avoid in enterprise Azure environments.
