DNS, Private Link e Segurança de rede

A topologia conecta; o DNS faz os nomes resolverem para os lugares certos; a segurança controla quem fala com quem. Esta Parte 5 aprofunda os dois temas que mais quebram (ou salvam) arquiteturas privadas: DNS híbrido com o DNS Private Resolver e a segurança em camadas — fechando com Zero Trust aplicado à rede.

Resumo executivo

ℹ
DNS é a maior fonte de erro em arquiteturas privadas no Azure: um Private Endpoint sem a Private DNS Zone certa resolve para o IP público e a privacidade evapora. O DNS Private Resolver substitui as antigas VMs de forwarder, conectando o DNS do Azure ao on-premises nos dois sentidos. Na segurança, pense em camadas: segmentação (VNet/subnet/NSG/ASG), inspeção centralizada (Azure Firewall/NVA), proteção de aplicação (WAF no Application Gateway ou Front Door), acesso privado a PaaS (Private Endpoints) e controle de egress. E lembre: rede privada não é Zero Trust — Zero Trust exige identidade, privilégio mínimo, inspeção e validação contínua, não só IP privado.
⚠
Recursos de cada SKU (ex.: capacidades premium do Azure Firewall) e comportamentos de DNS mudam — valide na documentação oficial antes de desenhar.

DNS híbrido e o DNS Private Resolver

O Azure DNS Private Resolver resolve o DNS híbrido sem VMs de forwarder. Ele tem inbound endpoints (o on-premises consulta nomes privados do Azure) e outbound endpoints (workloads do Azure encaminham consultas ao DNS on-premises, via regras de conditional forwarding). O fluxo típico:

  • 1 — Cliente on-premises consulta seu DNS local.
  • 2 — O DNS local encaminha a zona privada do Azure para o inbound endpoint do Private Resolver.
  • 3 — Workloads no Azure consultam o DNS configurado na VNet.
  • 4 — O Private Resolver encaminha (conditional forwarding) o que for on-premises ao DNS local pelo outbound endpoint.
  • 5 — O Private Endpoint resolve para o IP privado via Private DNS Zone (Parte 1).

Armadilhas comuns de DNS

  • Criar Private Endpoint sem configurar DNS — o FQDN resolve para o IP público.
  • VNet não linkada à Private DNS Zone do serviço.
  • DNS on-premises não encaminha as zonas privadas do Azure.
  • Duplicar Private DNS Zones por assinatura sem governança (centralize ou padronize).
  • Esquecer que o FQDN continua o mesmo — só a resolução muda para o IP privado.
  • Misturar DNS customizado e Azure-provided DNS sem desenho claro.

Firewall, WAF e controle de egress/ingress

Não confunda os papéis: o Azure Firewall inspeciona tráfego de rede (norte-sul e leste-oeste); o WAF protege aplicações web — no Application Gateway (proteção regional) ou no Azure Front Door (proteção global na borda). O controle de egress (saída) costuma passar pelo firewall com regras de FQDN; o ingress (entrada) por Application Gateway/Front Door com WAF. Inspeção norte-sul é o tráfego que entra/sai; leste-oeste é entre workloads (spoke-to-spoke da Parte 2).

Zero Trust aplicado à rede

⚠
Rede privada não é sinônimo de Zero Trust. Zero Trust pressupõe identidade forte, privilégio mínimo, segmentação, inspeção, monitoramento, validação contínua, redução de exposição pública e controle de egress. A rede é uma camada — combine com identidade (Entra ID) e governança (Parte 6).

Checklist

  • DNS — Private DNS Zones centralizadas/governadas e VNets corretas linkadas?
  • Híbrido — On-premises resolve nomes privados do Azure e vice-versa (Private Resolver)?
  • Camadas — Segmentação + NSG + firewall + WAF + Private Endpoints no lugar?
  • Egress/Ingress — Saída inspecionada por firewall; entrada protegida por WAF?
  • Zero Trust — Identidade, privilégio mínimo e validação contínua, não só IP privado?
