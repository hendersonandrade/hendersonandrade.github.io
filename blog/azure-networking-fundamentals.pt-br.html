Fundamentos de Rede no Azure

Antes de escolher entre Hub-and-Spoke, Virtual WAN ou conectividade híbrida, é preciso dominar os blocos que compõem qualquer topologia no Azure: VNet, subnets, NSG, rotas, peering, gateway transit, firewall, Private Link e DNS. Esta é a Parte 1 de uma série sobre arquitetura de redes no Azure — a base conceitual sobre a qual as decisões corporativas das próximas partes se apoiam.

Resumo executivo

ℹ
A topologia de rede é uma das primeiras e mais duradouras decisões de uma Landing Zone Azure — difícil de mudar depois que dezenas de assinaturas e VNets já dependem dela. Esta parte estabelece os fundamentos: a VNet é o perímetro privado; subnets segmentam; NSG/ASG filtram; UDR redireciona o caminho do tráfego (tipicamente para um firewall); peering conecta VNets pelo backbone, mas não é transitivo; gateway transit permite que spokes usem o gateway híbrido do hub; Private Link traz PaaS para dentro da rede privada — e exige DNS bem planejado. As partes 2 a 4 mostram como combinar esses blocos em Hub-Spoke, Virtual WAN e conectividade híbrida; as partes 5 e 6 cobrem DNS/segurança e operação/governança/FinOps. Nenhum modelo é universalmente correto — o que muda é o trade-off entre controle, escala, custo e esforço operacional.
⚠
Capacidades, limites de SKU, disponibilidade regional e preços mudam com frequência. Toda vez que este artigo cita um comportamento específico, valide na documentação oficial da Microsoft Learn antes de desenhar para produção. Os números aqui são conceituais, não substituem a referência atual.

Por que a topologia de rede é uma decisão central

Rede no Azure não é "conectar VNets". A topologia define como workloads se comunicam, como aplicações acessam serviços PaaS, como o ambiente on-premises chega à nuvem, onde o tráfego é inspecionado, como a governança é aplicada e como o custo de conectividade cresce. É a quarta das oito áreas de design das Azure Landing Zones — e talvez a de maior inércia: trocar a topologia depois que a plataforma cresceu é caro e arriscado.

Por isso, uma boa arquitetura separa plataforma de workloads: os serviços compartilhados de conectividade (firewall, gateways, DNS, Bastion) vivem em uma assinatura de conectividade, governada pelo time de plataforma; os workloads aterrissam em assinaturas próprias com guardrails herdados. Essa decisão — central no Cloud Adoption Framework — só faz sentido se você entender os blocos abaixo.

Virtual Network e subnets

A Azure Virtual Network (VNet) é a unidade lógica de rede privada: um espaço de endereçamento (um ou mais blocos CIDR) dentro do qual você cria subnets. Recursos na mesma VNet se comunicam por padrão; o que cruza a fronteira da VNet depende de peering, gateways e rotas. O planejamento de IP é a decisão mais difícil de reverter: escolha blocos que não se sobreponham ao on-premises nem a outras VNets que um dia farão peering, e deixe espaço para crescer.

Algumas subnets têm nomes reservados exigidos por serviços específicos — planeje-as no desenho de IP desde o início:

Subnets também suportam delegação (entregar a subnet ao plano de controle de um serviço, como App Service VNet Integration, AKS ou bancos gerenciados). Planeje subnets dedicadas para Private Endpoints, integração de App Service e AKS — cada um tem requisitos próprios de tamanho.

Network Security Groups e ASGs

O Network Security Group (NSG) é um filtro stateful de 5-tupla aplicado a uma subnet ou a uma NIC. Regras têm prioridade, direção (inbound/outbound) e podem usar service tags (ex.: Storage, AzureKeyVault) em vez de IPs, e Application Security Groups (ASG) para agrupar NICs por papel (ex.: web, db) e escrever regras por grupo lógico, não por IP.

ℹ
NSG é necessário, mas não suficiente. Ele faz microssegmentação L3/L4, não inspeção de aplicação, FQDN, TLS ou IDS/IPS. Para isso você precisa de um firewall (próxima seções). Use NSG para o baseline de segmentação e firewall para a inspeção centralizada.

Rotas do sistema e User Defined Routes

O Azure cria system routes automaticamente (dentro da VNet, para a Internet, para prefixos de peering e de gateway). Você sobrescreve esse comportamento com User Defined Routes (UDR) em uma route table associada à subnet. O caso mais comum é a rota default 0.0.0.0/0 apontando para um Azure Firewall ou NVA — forçando todo o egress a passar por inspeção:

Forced tunneling (mandar até o egress de Internet para o firewall ou para o on-premises) é poderoso, mas não deve ser aplicado indiscriminadamente: aumenta latência, custo e o risco de quebrar dependências de plataforma. Use propagate gateway routes com consciência e valide as effective routes antes de confiar no desenho.

VNet Peering e Gateway Transit

O VNet Peering conecta duas VNets pelo backbone da Microsoft (regional ou global, entre regiões), com baixa latência e sem gateway. O ponto que mais causa confusão: peering não é transitivo. Se o spoke A faz peering com o hub e o spoke B também, A não fala com B automaticamente — é preciso roteamento explícito (via NVA/ firewall no hub, ou peering direto, ou Virtual WAN). As flags que governam isso: allow forwarded traffic, allow gateway transit e use remote gateway.

ℹ
Gateway transit resolve um problema clássico: o gateway híbrido (VPN/ExpressRoute) é caro e fica no hub. Com allow gateway transit no hub e use remote gateway no spoke, os spokes usam o gateway do hub sem ter um gateway próprio. Cuidados: um spoke usa um gateway remoto por vez, e a propagação de rotas precisa ser planejada para o on-premises enxergar os spokes e vice-versa.

Azure Firewall vs. NVA de terceiros

A inspeção centralizada — o "ponto único" por onde o tráfego norte-sul e leste-oeste passa — pode ser feita com o Azure Firewall (PaaS nativo, gerenciado, escala automática, regras de rede e de aplicação, FQDN tags, Threat Intelligence) ou com uma Network Virtual Appliance (NVA) de mercado (Palo Alto, Fortinet, Check Point...). A escolha é menos sobre features isoladas e mais sobre quem opera e o que já existe na organização:

CritérioAzure FirewallNVA de terceiros
OperaçãoGerenciado pela MicrosoftVocê opera HA, scaling, patching
Integração AzureNativa (logs, Policy, vWAN)Boa, mas via configuração
EscalaAutomáticaDimensionada por você
LicenciamentoIncluso no serviçoBYOL / marketplace
Funcionalidades avançadasIDPS/TLS em SKU específico (validar)Recursos do fornecedor
Melhor quandoQuer simplicidade e integraçãoJá tem padrão corporativo na NVA
⚠
Rede privada e firewall não são sinônimo de Zero Trust, e ExpressRoute não substitui criptografia ou controle de identidade. Segurança de rede é uma camada — não a única.

Private Link, Private Endpoints e DNS

O Private Endpoint (PE) traz um serviço PaaS (Storage, SQL, Key Vault...) para dentro da sua rede com um IP privado, eliminando a exposição pública. O detalhe que mais quebra ambientes: o FQDN do serviço continua o mesmo, mas precisa resolver para o IP privado — e isso depende de DNS bem configurado (uma Azure Private DNS Zone ligada à VNet, com o registro do PE). DNS mal feito resolve para o endpoint público e o PE vira um IP que ninguém usa.

Há ainda os Service Endpoints, mais antigos: dão à subnet uma rota otimizada para o serviço PaaS pela backbone, mas sem IP privado e sem a granularidade do Private Link. Em arquiteturas restritivas, o Private Endpoint costuma ser o padrão preferido:

NAT Gateway, Bastion, DDoS e Network Watcher

  • NAT Gateway — egress de saída com IP público estável e gerenciado para uma subnet, sem expor entrada. Diferente do firewall: NAT Gateway não inspeciona, só faz SNAT de saída.
  • Azure Bastion — acesso administrativo (RDP/SSH) às VMs sem IP público nas VMs e sem expor essas portas à Internet. Vive na AzureBastionSubnet.
  • DDoS Protection — proteção contra ataques volumétricos para VNets com recursos críticos expostos publicamente. Avalie em ambientes com superfície pública relevante.
  • Network Watcher — o canivete de troubleshooting: next hop, effective routes, effective security rules, connection troubleshoot, packet capture e topologia. Sem observabilidade, a rede vira caixa-preta (a Parte 6 aprofunda).

Armadilhas comuns (fundamentos)

  • Sobrepor endereçamento entre Azure e on-premises (ou entre VNets que farão peering) — quebra o roteamento e é caríssimo de corrigir depois.
  • Assumir que peering é transitivo e esperar que spokes conversem entre si sem rota.
  • Criar Private Endpoint sem configurar a Private DNS Zone — o FQDN resolve para o IP público e o PE não é usado.
  • Aplicar forced tunneling indiscriminado e quebrar dependências de plataforma, ou criar latência/custo desnecessários.
  • Tratar NSG como único controle de segurança, sem inspeção de aplicação/FQDN.
  • Não validar as effective routes antes de confiar no desenho de roteamento.

Checklist de fundamentos

  • Endereçamento — Plano de IP sem sobreposição com on-premises e com espaço para crescer?
  • Subnets — Subnets reservadas (Gateway/Firewall/Bastion) e dedicadas (PE/AKS/App Service) planejadas?
  • Segmentação — NSG/ASG no baseline e firewall/NVA para inspeção centralizada?
  • Roteamento — UDRs documentadas, propagação controlada, effective routes validadas?
  • Conectividade — Peering e gateway transit desenhados sabendo que peering não é transitivo?
  • PaaS privado — Private Endpoints com Private DNS Zones ligadas às VNets corretas?
  • Operação — Network Watcher e logs habilitados desde o dia 1?
