Azure + Terraform + Infracost: IaC enterprise com FinOps no pipeline

Um laboratório de ponta a ponta: dos conceitos de Infraestrutura como Código e FinOps até uma estrutura modular e profissional de Terraform no Azure — com backend de state provisionado por Bicep, pipeline GitHub Actions autenticado por OIDC sem segredos e estimativa de custo do Infracost comentada automaticamente em cada Pull Request. Todo o código vive num repositório companion que você pode clonar e rodar.

Henderson Andrade Publicado em 18 jun 2026 ~24 min de leitura Terraform · Azure · Infracost · GitHub Actions
Terraform FinOps · Infracost Bicep backend OIDC / WIF GitHub Actions 6 diagramas
ℹ
Pré-requisitos: uma subscription Azure (Owner/Contributor), Azure CLI, Terraform CLI ≥ 1.7, um repositório no GitHub e uma conta gratuita no Infracost. Este artigo foca nos conceitos e nas decisões de arquitetura; o passo a passo completo para colocar a esteira de pé — bootstrap, OIDC, Variables/Secrets, abrir PR, aplicar e destruir — está no README do repositório azure-iac-with-tf-infracost, com checklist de pré-requisitos e solução de problemas.

O que é, de fato, Infraestrutura como Código

Infraestrutura como Código (IaC) é a prática de descrever sua infraestrutura — redes, máquinas, bancos, permissões — em arquivos de texto versionados, e deixar uma ferramenta provisioná-la por você. Em vez de clicar no portal e torcer para lembrar o que fez, você declara o estado desejado e a ferramenta reconcilia a realidade com esse alvo.

Quatro conceitos sustentam tudo o que vem depois:

  • Declarativo > imperativo. Você descreve o que quer (uma VM Standard_B2s nesta subnet), não os passos para chegar lá. O Terraform calcula o como — criar, atualizar ou destruir — comparando o desejado com o real.
  • Idempotência. Rodar o mesmo código dez vezes leva ao mesmo resultado. Se nada mudou no código, apply não faz nada. Isso torna a automação segura de repetir.
  • State (estado). O Terraform mantém um arquivo de state que mapeia cada recurso do código para o recurso real no Azure. É a "fonte da verdade" da ferramenta — e por isso precisa ser guardado de forma remota, segura e com lock (voltaremos a isso).
  • Drift & imutabilidade. Drift é quando alguém muda algo à mão e a realidade diverge do código. O plan detecta o drift; o apply o corrige. A mentalidade imutável prefere recriar a remendar — servidores são gado, não bichinhos de estimação.

Por que isso importa. IaC traz repetibilidade (o ambiente de dev é idêntico ao de prod), versionamento (cada mudança é um commit com autor e motivo), revisão (infra muda via Pull Request, com revisão de pares), velocidade (subir um ambiente novo é rodar um pipeline) e governança (políticas e padrões viram código, não tribal knowledge). É a oitava área de design das Azure Landing Zones — automação de plataforma — levada ao nível do workload.

FinOps e o problema do custo invisível

Na nuvem, cada linha de IaC tem um preço. Trocar Standard_B2s por Standard_D8s_v5 é um diff de uma palavra que multiplica a fatura por dez. O problema é que, no fluxo tradicional, ninguém vê esse custo até a fatura chegar — quando o recurso já rodou o mês inteiro.

FinOps é a disciplina que traz responsabilidade financeira para a velocidade da nuvem: engenharia, finanças e negócio colaborando para que cada time veja e seja dono do próprio custo. A FinOps Foundation a organiza em três fases iterativas:

FasePergunta que respondeNo nosso lab
InformQuanto custa e de quem é?Infracost mostra o custo do diff no PR; tags identificam o dono.
OptimizeOnde dá para economizar?Comparar SKUs no plan antes de aplicar; right-sizing.
OperateComo manter sob controle?Guardrails/políticas do Infracost que falham PRs caros.

A grande ideia é shift-left de custo: empurrar a visibilidade financeira para a esquerda da esteira — para o Pull Request, onde a mudança ainda é barata de ajustar — em vez de descobri-la na fatura, onde já é tarde.

Infracost a fundo

A CLI do Infracost é open-source (Apache 2.0) e estima o custo mensal da sua infraestrutura Terraform antes de aplicá-la. Ela lê o seu código (ou o plano gerado), identifica cada recurso cobrável, consulta a base de preços das nuvens e devolve um breakdown (custo total) ou um diff (quanto este PR adiciona ou remove). Vale a distinção: a CLI é open source, mas faz parte de um produto comercial — a estimativa usa a Cloud Pricing API hospedada (gratuita, exige chave/registro) e o Infracost Cloud é um SaaS comercial (com camada gratuita).

Como funciona, por dentro:

  1. Parse. O Infracost lê o projeto Terraform (HCL) ou um terraform plan em JSON e extrai os recursos e seus parâmetros relevantes para preço (SKU, tier, região, tamanho de disco).
  2. Pricing. Para cada recurso, ele consulta a Cloud Pricing API do Infracost — um catálogo dos preços públicos de AWS, Azure e GCP — e calcula o custo mensal projetado.
  3. Output. Emite uma tabela (breakdown) ou um comparativo (diff) em texto, JSON, HTML ou como comentário de Pull Request.

Infracost Cloud é o painel SaaS (com camada gratuita) que agrega esses números entre repositórios, mostra tendências e, principalmente, permite definir guardrails/políticas — por exemplo, "marcar para revisão qualquer PR que aumente o custo em mais de US$ 100/mês". As vantagens são claras: custo visível no PR, decisões de SKU informadas, prevenção de surpresas na fatura e cultura FinOps embutida na esteira. As limitações a ter em mente: é uma estimativa de preço de lista (não considera descontos de Reserved Instances/Savings Plans nem uso real por hora), e recursos com custo por consumo (egress, requisições) dependem de premissas de uso que você pode ajustar no infracost-usage.yml.

A estrutura enterprise de IaC

Um main.tf de 800 linhas funciona até o dia em que dois times precisam mexer nele. A abordagem enterprise é modular: módulos pequenos e reutilizáveis, uma raiz que os compõe, e variáveis por ambiente. Esta é a árvore do repo companion:

azure-iac-with-tf-infracost/
# bootstrap do backend de state — Bicep (execução única)
bootstrap/
  main.bicep            # RG + Storage Account + container "tfstate"
  modules/state-storage.bicep
  deploy.sh

# raiz Terraform aplicada pelo pipeline
infra/
  providers.tf         # versões + backend "azurerm" remoto (OIDC)
  main.tf              # compõe os módulos
  variables.tf  outputs.tf  data.tf
  dev.tfvars    prod.tfvars

# módulos reutilizáveis (1 responsabilidade cada)
modules/
  networking/          # VNet + subnet + NSG
  storage-account/     # 1 Storage Account
  app-service/         # 1 App Service Plan + Web App

infracost/infracost.yml
.github/workflows/terraform.yml

Cada módulo expõe um contrato claro — variables.tf (entradas), main.tf (recursos) e outputs.tf (saídas) — e nada mais. A convenção de nomes é <recurso>-<workload>-<environment> (ex.: vnet-iacdemo-dev), e tags comuns (environment, managedBy, costCenter) são mescladas em todos os recursos — essenciais para o FinOps atribuir custo ao dono certo.

Por que o tfstate é provisionado via Bicep

O Terraform guarda o state num backend. Em produção, esse backend é remoto — no Azure, uma Storage Account + container — para que o state seja compartilhado pelo time e pelo pipeline, fique criptografado e tenha lock (impede dois apply simultâneos corromperem tudo).

Mas aí surge o problema do ovo e da galinha: o Terraform precisa do backend para rodar, mas o backend é... infraestrutura. Se você criar a Storage Account com Terraform, onde mora o state que descreve essa Storage Account? Num backend local descartável, que ninguém mais tem? Num state que se referencia a si mesmo? Qualquer caminho é frágil.

ℹ
A solução limpa é fazer o bootstrap do backend com Bicep. O Bicep é nativo do Azure (compila para ARM), é idempotente e — crucial — não guarda state local: o "state" dele é o próprio Resource Manager. Você roda o Bicep uma vez para criar o backend, e a partir daí todo o Terraform usa esse backend. O bootstrap fica fora do ciclo de vida do Terraform, exatamente onde deve estar.
bootstrap/modules/state-storage.bicep
resource sa 'Microsoft.Storage/storageAccounts@2023-05-01' = {
  name: storageAccountName
  location: location
  sku: { name: 'Standard_LRS' }
  kind: 'StorageV2'
  properties: {
    minimumTlsVersion: 'TLS1_2'
    allowBlobPublicAccess: false
    supportsHttpsTrafficOnly: true
  }
}

resource blobService 'Microsoft.Storage/storageAccounts/blobServices@2023-05-01' = {
  parent: sa
  name: 'default'
  properties: {
    isVersioningEnabled: true          // recupera state sobrescrito
    deleteRetentionPolicy: { enabled: true, days: 30 }
  }
}
ℹ
No repositório companion, o bootstrap roda por um workflow dedicado (bootstrap-tfstate.yml) com a mesma autenticação OIDC sem segredos — disparado em push na main ou sob demanda. O az deployment sub create é idempotente: rodar de novo mantém o backend existente. Os valores do backend (resourceGroupName e storageAccountName) ficam em bootstrap/main.parameters.json, a fonte única de verdade que o pipeline lê para configurar o terraform init.

Workload Identity Federation: pipeline sem segredos

O jeito antigo de autenticar um pipeline no Azure era criar um service principal com client secret e guardar essa senha como secret do GitHub. Problema: é uma credencial de longa duração que vaza, expira e precisa de rotação. O jeito moderno é OpenID Connect (OIDC), também chamado de Workload Identity Federation.

A ideia: o GitHub Actions emite um token OIDC de curtíssima duração para cada execução, provando "sou o repositório X, no branch Y". Você configura no Entra ID um Federated Credential que confia nesse token para um subject específico (repo:org/repo:ref:refs/heads/main). O Azure troca o token do GitHub por um access token — nenhuma senha é armazenada em lugar nenhum.

criar o federated credential (Azure CLI)
az ad app federated-credential create --id "$appId" --parameters '{
  "name": "gh-main",
  "issuer": "https://token.actions.githubusercontent.com",
  "subject": "repo:hendersonandrade/azure-iac-with-tf-infracost:ref:refs/heads/main",
  "audiences": ["api://AzureADTokenExchange"]
}'
ℹ
Na prática você cria uma credencial federada por gatilho, porque o subject muda: ...:ref:refs/heads/main (push), ...:pull_request (PRs) e ...:environment:production (o job de apply protegido pelo environment). Quando um job declara environment: production, o GitHub troca o subject da branch pelo do environment — por isso o apply precisa da sua própria credencial. Os três comandos estão no README.

Os samples Terraform, comentados

O providers.tf fixa versões e declara o backend remoto. Note o use_oidc = true — é ele que conecta tudo ao login sem segredos da seção anterior. Os valores do backend que variam (resource group, storage account) não podem usar variáveis, então são passados no init via -backend-config:

infra/providers.tf
terraform {
  required_version = ">= 1.7.0"
  required_providers {
    azurerm = { source = "hashicorp/azurerm", version = "~> 4.0" }
  }
  backend "azurerm" {
    container_name = "tfstate"
    key            = "azure-iac-with-tf-infracost.tfstate"
    use_oidc       = true   # backend autentica com o token OIDC
  }
}

provider "azurerm" {
  features {}
  use_oidc = true           # sem secret: login via WIF
}

Um módulo segue sempre o mesmo contrato. O de storage usa um sufixo aleatório para garantir nome único global:

modules/storage-account/main.tf
resource "azurerm_storage_account" "this" {
  name                            = local.sa_name
  resource_group_name             = var.resource_group_name
  location                        = var.location
  account_tier                    = var.account_tier      # Standard
  account_replication_type        = var.replication_type  # LRS
  min_tls_version                 = "TLS1_2"
  allow_nested_items_to_be_public = false
  https_traffic_only_enabled      = true
  tags                            = var.tags

  blob_properties {
    versioning_enabled = true
  }
}

E a raiz compõe tudo: injeta o nome do resource group que ela mesma criou (azurerm_resource_group.this.name) como entrada de cada módulo, e usa um data source (azurerm_subscription) para ler metadados sem criar nada:

infra/main.tf · data.tf
# data.tf — lê o que já existe, não cria nada
data "azurerm_client_config" "current" {}
data "azurerm_subscription" "current" {}

# main.tf — o RG criado pela raiz vira entrada dos módulos
resource "azurerm_resource_group" "this" {
  name     = "rg-${local.name_suffix}"
  location = var.location
}

module "storage" {
  source              = "../modules/storage-account"
  resource_group_name = azurerm_resource_group.this.name   # saída → entrada
  location            = var.location
  # ...
}

O pipeline GitHub Actions, passo a passo

O workflow tem três jobs: bootstrap (em push na main, garante que o backend de state exista chamando o workflow bootstrap-tfstate.yml antes do init), plan (roda em todo PR e push, valida e estima custo) e apply (só em push na main, atrás de um environment com aprovação manual). Vamos pelos blocos.

1 · Triggers e permissões. O bloco permissions é o coração do OIDC: id-token: write autoriza o job a pedir o token. pull-requests: write deixa o Infracost comentar no PR.

.github/workflows/terraform.yml — gatilho & permissões
on:
  pull_request: { branches: [main] }
  push:         { branches: [main] }

permissions:
  id-token: write        # pedir o token OIDC
  contents: read         # checkout do código
  pull-requests: write   # comentar custo no PR

2 · Login OIDC + init com backend. A action azure/login recebe só os identificadores (client/tenant/subscription) — nenhuma senha — e usa o token OIDC. Um passo anterior lê resourceGroupName e storageAccountName de bootstrap/main.parameters.json (com jq) e os exporta como TF_BACKEND_RG/TF_BACKEND_SA; em seguida o init os passa ao backend via -backend-config. Assim o parameters.json é a fonte única de verdade — sem duplicar esses valores nas Variables do GitHub.

.github/workflows/terraform.yml — login & init
- name: Azure login (OIDC)
  uses: azure/login@v2
  with:
    client-id:       ${{ vars.AZURE_CLIENT_ID }}
    tenant-id:       ${{ vars.AZURE_TENANT_ID }}
    subscription-id: ${{ vars.AZURE_SUBSCRIPTION_ID }}

- name: Load backend configuration   # lê o parameters.json com jq
  run: |
    p=../bootstrap/main.parameters.json
    echo "TF_BACKEND_RG=$(jq -er '.parameters.resourceGroupName.value' $p)" >> $GITHUB_ENV
    echo "TF_BACKEND_SA=$(jq -er '.parameters.storageAccountName.value' $p)" >> $GITHUB_ENV

- name: Terraform init
  run: |
    terraform init \
      -backend-config="resource_group_name=${TF_BACKEND_RG}" \
      -backend-config="storage_account_name=${TF_BACKEND_SA}"

3 · fmt → validate → plan. Três portões em ordem crescente de custo: formatação (estilo), validação (sintaxe e referências) e o plan, que vira um artefato reutilizado pelo apply — garantindo que o que foi revisado é o que será aplicado.

.github/workflows/terraform.yml — validação & plan
- run: terraform fmt -check -recursive
- run: terraform validate
- run: terraform plan -input=false -var-file="dev.tfvars" -out=tfplan

- name: Upload plan artifact
  uses: actions/upload-artifact@v4
  with: { name: tfplan, path: infra/tfplan }

4 · Apply protegido. O job apply só dispara em push na main e declara environment: production — o que ativa, no Settings → Environments do GitHub, a exigência de revisores aprovarem antes do deploy. É o seu portão humano sobre a automação.

Integrando o Infracost no pipeline

Com o pipeline de pé, plugar o Infracost são três passos, só no contexto de Pull Request: instalar a CLI, gerar o diff de custo e postar/atualizar o comentário no PR.

.github/workflows/terraform.yml — Infracost no PR
- name: Setup Infracost
  if: github.event_name == 'pull_request'
  uses: infracost/actions/setup@v3
  with: { api-key: ${{ secrets.INFRACOST_API_KEY }} }

- name: Infracost diff
  if: github.event_name == 'pull_request'
  run: |
    infracost diff \
      --config-file=../infracost/infracost.yml \
      --format=json --out-file=/tmp/infracost.json

- name: Comentar no PR
  if: github.event_name == 'pull_request'
  run: |
    infracost comment github \
      --path=/tmp/infracost.json \
      --repo=$GITHUB_REPOSITORY \
      --github-token=${{ secrets.GITHUB_TOKEN }} \
      --pull-request=${{ github.event.pull_request.number }} \
      --behavior=update

O --behavior=update mantém um único comentário que é atualizado a cada push no PR — em vez de poluir a conversa com um novo a cada commit. No repositório companion o pipeline vai um pouco além: gera um baseline a partir do código da main e usa --compare-to para um diff preciso, publica a tabela detalhada também no Summary do run (GITHUB_STEP_SUMMARY) e custa tanto dev quanto prod via infracost/infracost.yml.

ℹ
Moeda e outras customizações. Por padrão os custos saem em USD (as nuvens publicam em dólar; a CLI converte pela cotação atual). Para exibir em BRL ou qualquer código ISO 4217, defina INFRACOST_CURRENCY: BRL no bloco env: do workflow (ou infracost configure set currency BRL localmente). Premissas de uso por consumo vão num infracost-usage.yml. O README detalha essas e outras opções.
ℹ
Guardrails. No Infracost Cloud (camada gratuita) você liga políticas que marcam ou bloqueiam PRs acima de um limite — ex.: "aumento > US$ 100/mês exige aprovação do FinOps". É a fase Operate do FinOps virando portão automático na esteira.
⚠
É um laboratório que gera custo real (App Service ≈ US$ 13–15/mês em dev). Ao terminar, rode terraform destroy -var-file=dev.tfvars e, se não for reutilizar, apague o RG do state com az group delete. O próprio Infracost te lembra de quanto está em jogo.
ℹ
Mãos à obra. O README do repositório traz o passo a passo numerado de ponta a ponta — clonar, bootstrap do state, criar a identidade OIDC, configurar Variables/Secrets, abrir o primeiro PR e ver o custo, aprovar o apply e destruir o lab. Use este artigo para entender o porquê; use o README para o como.

Checklist de produção

  • State — Backend remoto no Azure Storage, com versionamento e lock, criado por bootstrap Bicep separado?
  • Autenticação — Pipeline via OIDC/Workload Identity Federation, sem nenhuma client secret guardada?
  • Modularidade — Recursos em módulos reutilizáveis com contrato claro (variables/outputs), não num monólito?
  • Ambientes — Configuração por ambiente isolada em *.tfvars (dev/prod), mesmo código?
  • Qualidade na CI — fmt -check, validate e plan rodando em todo PR, com o plan virando artefato?
  • Portão humano — apply só na main, atrás de um environment com aprovação obrigatória?
  • FinOps — Infracost comentando o diff de custo no PR, com guardrail de limite no Infracost Cloud?
  • Tags — environment, managedBy e costCenter em todos os recursos para atribuição de custo?
  • Ciclo de vida — Caminho de destroy documentado para ambientes efêmeros não virarem custo esquecido?
